# 专利撰写 Skill

适用场景：基于已有工业项目成果（如换热器温度预测、设备健康管理等），撰写数据分析/AI预测方向的发明专利。

## 核心原则

1. **先检索再选题** — 了解现有专利布局，避免核心技术重合
2. **项目为基、创新为翼** — 以已验证的项目成果为基础，叠加增量创新提升创造性
3. **不删用户文件** — 原始资料、中间数据均保留，不主动清理任何文件
4. **结构化输出** — 每一步产出 md 文档，全程可追溯
5. **交底书 md 为源、Word 用脚本对齐** — 本仓库以 **`技术交底书_示例.md`** + **`_convert_技术交底书_示例_md_to_docx.py`** 演示整条链路；正式稿可另存 md、复制并改写脚本中的路径与版式规则以对接你的文件
6. **Skill 定义规则、Tool 强制验收** — 写作要求不能只依赖一次提示词；每个章节生成或修改后必须调用质量审查 tool，不通过则自动修订并复检
7. **外部检索只做避让，不做创新来源** — AnySearch/网页/相似专利结果只能用于补充背景、识别已有技术、提取不可复用特征和检查重合风险；不得把外部已有专利的核心方案直接写成新专利创新点

## 工作流程

```
1. 解析项目文档 → 提取技术方案全貌（文字/_doc_content.txt、表格/_doc_tables.txt）
2. 行业专利检索 → 识别现有技术空白
3. 挖掘创新点 → 提出差异化选题方向（≥10个）
4. 可专利性评估 → 评估新颖性/创造性/实用性
5. 撰写技术交底书 → 完整初稿（含权利要求、附图说明）
6. 附图 → Mermaid 维护于 `05_附图/专利附图全集_示例.md`（示例清单），导出 PNG 入 `05_附图/专利一/`（示例：`01_流程简图_示例.png`）
7. md→Word → **`_convert_技术交底书_示例_md_to_docx.py`**（SimHei/SimSun、版心插图缩放）
8. 授权难度评估 → 对比分析各篇专利的授权概率
```

## 1. 项目文档解析

### Word文档提取

```python
import docx, os

doc = docx.Document("项目文档.docx")
lines = [p.text for p in doc.paragraphs]
open("_doc_content.txt", "w", encoding="utf-8").write("\n".join(lines))

tables = []
for i, t in enumerate(doc.tables):
    rows = [" | ".join([c.text.strip() for c in r.cells]) for r in t.rows]
    tables.append(f"=== Table {i} ===\n" + "\n".join(rows))
open("_doc_tables.txt", "w", encoding="utf-8").write("\n\n".join(tables))
```

需要依赖：`python-docx`（可用 `uv add python-docx`，见项目 `docs/UV环境说明.md`）

提取后梳理以下关键信息：
- 系统架构（数据流、模块划分）
- 各模块使用的算法/模型
- 输入数据（类型、来源、频率）
- 预测精度指标（MAE、RMSE、R²）
- 已有的创新设计（特殊的特征工程、模型组合策略等）

## 2. 行业专利检索

### 检索策略

分三轮检索，中英文交替：

1. **中文检索**：`[设备类型] + [预测目标] + 专利 + 机器学习/深度学习`
2. **英文检索**：`[device type] + [prediction target] + patent + machine learning`
3. **细分技术检索**：`[设备类型] + [具体方法，如迁移学习/因果推断/图神经网络] + 专利`

### 大模型检索与人工官方库（必做补充）

- **大模型/联网检索**可作为快速摸底与关键词扩展，但**不能替代**对**中国发明专利/实用新型**的正式库检索，易遗漏或滞后。
- **务必提醒申请人/撰写人**在 **[国家知识产权局专利检索及分析系统 · 常规检索](https://pss-system.cponline.cnipa.gov.cn/conventionalSearch)** 上，用本节「检索策略」中的**中英文关键词**（及同义词、IPC/CPC 分类号）**人工检索**，核对权利要求与说明书要点，**降低遗漏相近现有技术的风险**。
- 对外部检索结果（含大模型汇总）应标注检索日期与检索式，便于后续答复审查意见时追溯。

### 梳理输出格式

建立"已有核心技术表"，后续选题必须避开：

```markdown
| 已有方向 | 核心技术 | 代表专利/论文 |
|---------|---------|-------------|
| 时序预测 | LSTM/CNN-LSTM | CN1xxxxx |
| 性能预测 | ANN预测传热系数 | US20240xxxxx |
```

## 3. 创新点挖掘

### 核心思路：跨行业方法迁移

从其他行业借鉴成熟的数据分析方法，迁移到目标领域：

| 来源行业 | 方法 | 迁移方向 |
|---------|------|---------|
| 金融风控/医疗 | 共形预测（Conformal Prediction） | 从点预测升级为区间预测+可信度量化 |
| 经济学/流行病学 | 因果推断（Causal Inference） | 从相关性建模升级为因果性建模 |
| 电商推荐/数据流 | 概念漂移检测（Concept Drift） | 从固定周期重训练升级为漂移驱动自适应 |
| 通信/NLP | VMD + 多任务学习 | 从整体建模升级为分频建模+多目标协同 |
| 生物医学/可靠性 | 生存分析（Survival Analysis） | 从预测数值升级为预测事件时间 |
| 自动驾驶/传感器 | 多源数据质量评估 | 从固定权重升级为质量驱动动态权重 |
| 计算机视觉/NLP | 迁移学习 / 对比学习 | 少样本快速建模 |
| 气象预报 | 不确定性级联传播 | 全链路概率预测 |
| AutoML/XAI | 自动特征工程 + SHAP | 特征搜索+可解释性闭环 |

### 选题输出要求

每个选题必须包含：
- 跨行业来源
- 创新点描述（与现有技术的本质差异）
- 可结合的项目基础
- 数据分析强度/实施难度/专利壁垒评估

## 4. 可专利性评估

### 评估三要件

| 要件 | 评估方法 |
|------|----------|
| 新颖性 | 专利库检索是否有相同技术方案 |
| 创造性 | 判断技术方案是否"非显而易见"——不是已知方法的简单组合 |
| 实用性 | 是否有实际运行数据支撑效果 |

### 先公开风险评估

- 项目交付物是否已公开？（涉密项目如核电通常不构成公开）
- 是否在学术会议/期刊发表过？
- 知识产权归属是否明确？（确认合同条款）

### 写作策略选择

```
纯项目路径：直接基于已实现的技术方案撰写
  └── 适用于：项目中有明确的原创方法（如"基准曲线+残差修正"）

项目+增量创新路径：项目基础 + 叠加1-2个创新方法
  └── 适用于：项目方法本身创造性一般，需要增强（如"已知算法的组合"）
```

## 5. 技术交底书结构

### 标准结构

```markdown
# [简短发明名称]

## 标题页（文档标题、文档类型、申请人/单位、发明人/作者、联系电话、邮箱）
## 一、发明名称
## 二、技术领域
## 三、背景技术
  - 只写本发明对应行业场景中的2-3个问题点
  - 不写泛泛行业综述
  - 不使用“技术空白一/二/三”“补足技术空白”等直白表述
## 四、发明内容
  ### 4.1 关键创新点（放在发明内容开头，不再单独另起“区别于现有技术”章节）
  ### 4.2 发明目的
  ### 4.3 拟解决的技术问题（逐条对应背景技术中的问题）
  ### 4.4 总体技术方案（步骤S1-Sn，含必要公式）
  ### 4.5 数据/环境/对象建模
  ### 4.6 输入特征/状态空间/任务上下文设计
  ### 4.7 约束机制/分配机制/控制机制
  ### 4.8 评价函数/评分机制/奖励机制
  ### 4.9 核心算法/模型训练/协同框架
  ### 4.10 在线部署与闭环流程
  ### 4.11 系统组成
  ### 4.12 有益效果（有证据才写量化结果；无证据时只做简洁定性说明）
## 五、保护范围（方法、系统、装置、存储介质及核心技术特征边界）
## 六、附图说明（列出需配图清单；与归档 PNG 文件名对照）
## 七、具体实施方式（含实际数据和精度指标；没有数据时标注待补充）
## 八、附图
```

### 本项目写作约束

- 标题必须短，只体现发明是什么，不把全部算法、指标和流程堆进标题；禁止使用“基于……的……”句式。
- 例如应写“铝电解槽阳极效应早期预警方法”，不要写“基于VMD频带能量特征与时序网络的铝电解槽阳极效应早期预警方法”。
- 背景技术必须服务于本专利的问题链：背景提出 a/b/c，发明内容对应解决 a/b/c。
- 不能直接把“行业没有该技术”“补了技术空白”作为创新论证。
- 有益效果不得编造百分比、金额、精度提升等量化数据；只有材料或检索结果支撑时才写量化结论。
- 全文避免重复出现多个“技术交底书”标题。
- 以上规则属于强制验收条件。章节生成后由 `patent_quality_tool.py` 执行确定性检查，失败时根据问题清单自动修订并复检。
- 最终导出前再次执行全文检查，确认标准章节、短标题、问题对应、有益效果证据和保护范围均已落实。
- 涉及公式时必须调用 formula-formatting skill：统一使用标准 LaTeX，行内 `$...$`、独立公式 `$$...$$`，并在公式后解释变量和单位。
- 最终专利文章只呈现发明主题内容。生成过程、质量自查、修复说明、提示词约束、skill/tool 调用、后端实现以及“未使用某句式/使用某规则”等说明只能保留在应用交互或历史记录中，不能写入 Markdown 正文或 Word 文档。

### 权利要求撰写要点

- 独立权利要求：覆盖核心方法步骤，不涉及具体算法名称（用"回归模型"而非"SVR"）
- 从属权利要求：细化具体实现（算法选型、参数范围、特征工程细节）
- 系统权利要求：按功能模块划分（数据采集模块、预测模块、评估模块等）
- 用"其特征在于"引导区别技术特征

### 插图与正文（推荐实践）

- 技术交底书 md 中插图路径指向 `05_附图/专利一/*.png`；图号按正文出现顺序连续，图题居中可用 HTML `<div align="center">` + `<img>`。
- **专利附图全集（示例）**：`05_附图/专利附图全集_示例.md`；正式撰写时可另存为 `专利附图全集.md` 并扩充。
- PNG 导出（三种并列）：① **本地开箱**：**Google Chrome** + `uv sync --extra figures` + `uv run python 02_资料分析/scripts/_simple_generate.py`（示例仓库仅映射 **`fig示例_流程简图_示例.mmd`**）；② **Node**：`npx @mermaid-js/mermaid-cli`；③ **mermaid.live**。详见 **`docs/UV环境说明.md` §5**。附图清单见 `05_附图/专利一/README.md`。

## 6. 授权难度评估

### 评估维度

| 维度 | 说明 |
|------|------|
| 新颖性评级 | 是否检索到相同/相近方案 |
| 创造性评级 | 审查员最可能的驳回理由是什么 |
| 答辩难度 | 技术点多寡、论证复杂度 |
| 授权概率 | 综合评级（★1-5） |
| 授权后价值 | 保护范围大小、是否容易被绕过 |

### 常见驳回理由与应对

| 驳回理由 | 应对策略 |
|---------|---------|
| "已知方法的简单组合" | 强调各模块之间的技术联动效果和整体性 |
| "属于本领域常规手段" | 用实验数据证明"技术效果显著" |
| "与对比文件X的区别不构成创造性" | 聚焦论证具体方案的特异性（参数设计、公式推导） |

## 7. md 转 Word（示例链路）

### 脚本（仓库 `02_资料分析/scripts/`）

| 脚本 | 用途 |
|------|------|
| **`_convert_技术交底书_示例_md_to_docx.py`** | **推荐入口（示例）**：占位正文 + **`01_流程简图_示例.png`**；SimHei/SimSun 四槽、Heading 后处理、**插图按版心缩放**（建议 Pillow）。输出 **`技术交底书_示例.docx`**。 |

若章节结构、插图命名与示例差异较大，可在同目录**新增**专用转换脚本（可复制本文件改名后改路径与解析逻辑）。

### 运行方式

```bash
uv run python 02_资料分析/scripts/_convert_技术交底书_示例_md_to_docx.py
```

### Word 版式要点（已在示例脚本中落实）

- 正文：**SimSun**，标题/表头/图题/加粗小标题：**SimHei**，`ascii`/`hAnsi`/`eastAsia`/`cs` 四槽写满，避免日文环境退化为 **MS Gothic**。
- 修补 **Heading N Char** 样式；保存前去掉标题 run 的 **`w:rStyle`** 再强制 SimHei。
- 插图：`add_picture(width=…)` 宽度由版心比例与图片像素高宽比自动计算，避免竖长流程图超出页面。
- 依赖：**python-docx** 必选；**Pillow** 强烈推荐（插图按比例缩放）。
